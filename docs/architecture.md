## 1. System design

```
[Caller] → POST /webhook/triage
              │
              ▼
       n8n workflow (Docker container, port 5678)
              │
              ├─ 1.  Validate Input
              ├─ 2.  Build Classify Prompt          ┐
              ├─ 3.  HTTP → Groq classify + enrich  │  one LLM call,
              │      (JSON mode, temperature 0)     │  returns 6 fields
              ├─ 4.  Parse + Route + Decide Escalation
              ├─ 5.  Build Summary Prompt           ┐
              ├─ 6.  HTTP → Groq summarize          │  second LLM call,
              │      (plain text, temperature 0.2)  │  3-sentence string
              ├─ 7.  Assemble Final Record ──┐
              │                              │
              ├─ 8.  Respond to Webhook ◄────┤  returns record to Caller
              │                              │
              └─ 9.  Switch by escalation ◄──┘
                       │
                       ├─► HTTP → Webhook.site/normal       (escalation.flag = false)
                       └─► HTTP → Webhook.site/escalation   (escalation.flag = true)
```

**Trigger.** An HTTP POST to the n8n webhook (`/webhook/triage`). One record in, one record out. The [scripts/send_samples.mjs](../scripts/send_samples.mjs) runner fans the five PDF samples into the webhook in sequence.

**State** lives in two deliberately separate stores:

| Store | Contents | Lifetime |
|---|---|---|
| `n8n_data` Docker volume | Live workflow + the encrypted Groq Header Auth credential | Persisted across container restarts. **Gitignored.** |
| `data/outputs/` in the repo | `results.json` (all 5 triaged records) + `escalations.json` (the flagged subset) | Versioned in git. Regenerated by re-running the script. |

The Webhook.site URLs are stand-ins for downstream team queues. Each request POSTs the canonical record to one of two paths (`/normal` or `/escalation`) so a reviewer can see both streams in the same dashboard.

**Why n8n in Docker.** One command — `docker compose up -d` — gives the reviewer the same orchestrator and same n8n version we built against. The committed workflow JSON (Phase 7) imports cleanly into any fresh n8n instance.

**Why Groq.** Free tier, sub-second latency, JSON mode support, no credit card. The runtime model is `llama-3.3-70b-versatile`. For the assessment's volume (5 records, ~10 LLM calls per full run) we never came close to a quota.

## 2. Routing logic

| Classifier output | Destination queue | Why |
|---|---|---|
| Bug Report | Engineering | Owns the fix even when only one user is affected |
| Incident/Outage | **Escalation (always)** | Auto-escalates regardless of confidence — see §3 |
| Feature Request | Product | Roadmap territory |
| Billing Issue | Billing *(unless dispute > $500)* | Most billing requests don't need a manager loop |
| Technical Question + auth keyword (`sso`, `saml`, `okta`, `auth`, `idp`, `oidc`) | IT/Security | Identity questions land with the team that owns identity |
| Technical Question, otherwise | Product | How-to / feasibility questions |
| Any category, confidence < 0.70 | **Escalation** | Fallback for low-confidence classifications |

Routing is **deterministic JavaScript in a Code node**, not a second LLM call. The LLM emits structured fields; the code maps those fields to a queue. Three reasons we split it that way:

1. **Auditability** — every routing decision is traceable to a specific `if` branch in version control
2. **Testability** — we can unit-test the routing rules without an LLM in the loop
3. **Cost** — the LLM is the bottleneck; not making it decide twice keeps latency under 3 s per record

The auth-keyword sub-route inside Technical Question (Sample #4: "set up SSO with Okta") was the smallest regex needed to handle the case cleanly. We considered a second LLM sub-classifier and rejected it: the keyword set is bounded, the cost of a miss is low (a Product person can re-route to IT/Security manually), and a second LLM call would have added ~1 s of latency for one branch of one category.

## 3. Escalation logic

Any one of these triggers the escalation flag. They are evaluated additively — every fired rule appends a string to `escalation.reasons[]` so a human reviewer can see exactly why a record was flagged:

1. Classifier confidence < 0.70 — the model itself is uncertain
2. Category == `Incident/Outage` — anything platform-wide always escalates regardless of confidence
3. Body contains the literal phrase `outage`
4. Body contains `down for all users`
5. Body contains `multiple users affected`
6. Category == `Billing Issue` AND the disputed amount > $500

**The billing rule deserves a note** — it's the only one that bit us. The PDF says "billing error > $500" — that's the *disputed* amount, not any number in the message. Our first pass naively flagged every dollar value, which mis-escalated Sample #3 ($1,240 invoice vs $980 contract → a $260 dispute). The fix: when the body mentions two or more dollar amounts, compute the dispute as `max(amounts) − min(amounts)`; when only one amount appears, treat it as the disputed amount. Sample #3 now stays in Billing where it belongs; a hypothetical "we were overcharged $600" would still escalate.

Final destination logic is straightforward: if `escalation.flag === true` the destination becomes `Escalation` regardless of category; otherwise the category's routing-table destination stands.

## 4. Production-scale trade-offs

**What we did, and the costs:**

| Decision | Chose | Cost |
|---|---|---|
| LLM split | One combined classify+enrich + one separate summarize | Extra network hop per record |
| Confidence source | Self-reported by the model | Confidence ≠ truth; documented in the prompt that ≥0.90 should be unambiguous-only |
| Webhook destinations | Hardcoded in HTTP nodes | URLs leak into the committed workflow.json; mitigated by treating them as ephemeral |
| Routing decisions | Deterministic JS, not LLM | We hand-code each rule (acceptable for ~6 rules; would not scale to 60) |
| Output store | Local JSON file + Webhook.site mirror | No SQL queryability; reviewer reads results.json directly from the repo |
| Orchestrator | n8n self-hosted in Docker | Single-node, no HA |
| Runtime LLM | Groq llama-3.3-70b-versatile only | Single-vendor dependency |

**What I'd change in production:**

*Reliability* — retry with exponential backoff on Groq 429/5xx (n8n's HTTP node supports this; left off for diagram clarity); dead-letter queue for records that fail validation or hit max retries; circuit breaker on Groq that fails over to OpenAI or Anthropic when Groq is unhealthy.

*Cost* — cache classifications keyed on a hash of the body (bursty incident reports during outages produce duplicates); move obvious incidents (matched by keyword pre-filter) to a non-LLM fast path before the LLM call; batch Groq calls when records arrive in the same window.

*Latency* — run the LLM classify and a keyword-based escalation pre-check in parallel; merge before routing. Pre-warm Groq with a periodic no-op if traffic is bursty (cold-start avoidance).

*Security / governance* — each downstream queue gets its own n8n credential (not a hardcoded URL); audit every escalation reason to a SIEM, not just Webhook.site; per-tenant scoping in the routing code (this assessment is single-tenant by design).

## 5. Phase 2 — one more week

| Idea | Why |
|---|---|
| Multi-tenant routing tables | Each ArcVault customer maps categories to queues differently; ingest from a config table instead of hardcoding |
| Active learning loop | When a human re-routes an escalated record, feed the corrected category back into a few-shot bank that updates the prompt nightly |
| Cohort analytics dashboard | Volume per category, escalation rate over time, median time-to-acknowledge per destination team |
| Real downstream integrations | Replace Webhook.site with actual Jira / Linear / Zendesk endpoints |
| Confidence calibration study | 100 hand-labeled cases, plot model confidence vs accuracy. If poorly calibrated, replace self-reported confidence with a logprob-based or trained calibrator |
