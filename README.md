# ArcVault AI Triage

> Valsoft Corporation — AI Engineer technical assessment, May 2026.
> Author: **Hadi Alkammouni**.

📹 **Loom walkthrough:** https://www.loom.com/share/8ba74ee489aa4cd89e17c99e573a1ef6

An AI-powered intake and triage pipeline for the synthetic B2B SaaS company **ArcVault**. Inbound customer messages (email, web form, support portal) hit an n8n webhook. The workflow classifies each request, pulls out the relevant entities, picks a destination queue, flags low-confidence or high-impact cases for a human, and writes a structured JSON record.

Run the quick-start below and you'll have the same five output records that are committed here, in under a minute.

## Quick start

```bash
# 1. Clone and enter
git clone https://github.com/Hadi-AlKammouni/arcvault-triage.git
cd arcvault-triage

# 2. Set up env (Groq key required; webhook URLs optional for re-running)
cp .env.example .env
# edit .env and paste your Groq API key (free tier at https://console.groq.com)

# 3. Bring up n8n
docker compose up -d
# n8n is now at http://localhost:5678 — first visit asks you to create an owner account

# 4. Import the workflow
# In n8n UI:  Workflows → Import from File → n8n/workflow.json
# Open the workflow → confirm the Groq Header Auth credential is wired
# Toggle the workflow to "Active" (top right)

# 5. Send all 5 samples through and regenerate the output records
node scripts/send_samples.mjs
```

That writes [data/outputs/results.json](data/outputs/results.json) and [data/outputs/escalations.json](data/outputs/escalations.json) — same files committed here, regenerated by your local run. The script also prints a one-line summary per sample so you can see latency, category, and destination right in the terminal.

**Prereqs:** Docker Desktop (or Docker Engine), Node 18+, a Groq API key (free).

## How the workflow is shaped

12 n8n nodes, three logical phases:

```
[Caller] → POST /webhook/triage
              │
              ├─ 1.  Validate Input
              ├─ 2.  Build Classify Prompt
              ├─ 3.  HTTP → Groq classify + enrich   (JSON mode, temperature 0)
              ├─ 4.  Parse + Route + Decide Escalation
              ├─ 5.  Build Summary Prompt
              ├─ 6.  HTTP → Groq summarize           (plain text, temperature 0.2)
              ├─ 7.  Assemble Final Record ──┐
              │                              │
              ├─ 8.  Respond to Webhook ◄────┤  returns record to Caller
              │                              │
              └─ 9.  Switch by escalation ◄──┘
                       │
                       ├─► HTTP → Webhook.site/normal      (not escalated)
                       └─► HTTP → Webhook.site/escalation  (escalated)
```

Classification and enrichment come from a single Groq call — combined for latency and cost (see [docs/prompt-rationale.md](docs/prompt-rationale.md)). The summary is a second, cheaper call so it can change without touching the classifier prompt. **Routing and escalation are deterministic JavaScript, not LLM calls**, so each decision traces back to a specific branch in version control.

For the full design — routing table, escalation rules, production trade-offs, Phase 2 plans — see [docs/architecture.md](docs/architecture.md).

## What I'd do differently with more time

Rough order of value:

1. **Add a `disputed_amount` field to the classifier output.** The routing code currently infers billing disputes with a regex + `max − min` heuristic. It worked on all 5 samples but it's brittle — having the LLM extract the dispute amount directly would kill the heuristic.
2. **Calibrate the self-reported confidence.** Hand-label ~100 cases, plot model confidence against accuracy. If it's miscalibrated, switch to logprobs or train a calibrator. The 0.70 escalation threshold fits the 5 samples but I haven't proven it at scale.
3. **Retry + dead-letter loop on Groq.** Fine without it for a free-tier demo; production wants exponential backoff and a DLQ for permanent failures.
4. **Per-destination summary personas.** Engineering summaries should lead with the error code, Billing with the invoice number, Product with the customer's intent. One prompt with branching wouldn't cost much.
5. **Active-learning loop.** When a human re-routes an escalated record, capture the correction and feed it back into the classifier's few-shot examples on a nightly job.

One thing that surprised me during the build: **the LLM was rarely the bug**. Sample #1 got misclassified once (fixed with an extra prompt rule and a few-shot example). Sample #3 looked like a classifier miss but turned out to be a bug in the routing JavaScript. Both stories are written up in [docs/prompt-rationale.md § "What the AI got wrong that I had to fix"](docs/prompt-rationale.md#what-the-ai-got-wrong-that-i-had-to-fix).
