# ArcVault AI Triage

> Valsoft Corporation — AI Engineer technical assessment, February 2026.
> Author: **Hadi Alkammouni**.

📹 **Loom walkthrough:** https://www.loom.com/share/8ba74ee489aa4cd89e17c99e573a1ef6

An AI-powered intake and triage pipeline for the synthetic B2B SaaS company **ArcVault**. Unstructured inbound customer messages (email, web form, support portal) arrive at an n8n webhook. The workflow classifies each request, extracts entities, decides a destination queue, flags low-confidence or high-impact cases for human review, and emits a structured JSON record per request.

A reviewer running the quick-start below gets the full pipeline locally in under a minute and reproduces the same five output records that are committed in this repo.

## Quick start

```bash
# 1. Clone and enter
git clone https://github.com/Hadi-AlKammouni/arcvault-triage.git
cd arcvault-triage

# 2. Set up env (Groq key required; webhook URLs optional for re-running)
cp .env.example .env
# edit .env and paste your Groq API key (free tier at https://console.groq.com)

# 3. Bring up n8n
docker compose up -d
# n8n is now at http://localhost:5678 — first visit asks you to create an owner account

# 4. Import the workflow
# In n8n UI:  Workflows → Import from File → n8n/workflow.json
# Open the workflow → confirm the Groq Header Auth credential is wired
# Toggle the workflow to "Active" (top right)

# 5. Send all 5 samples through and regenerate the output records
node scripts/send_samples.mjs
```

That writes [data/outputs/results.json](data/outputs/results.json) and [data/outputs/escalations.json](data/outputs/escalations.json) — the same files committed in this repo, regenerated by your local run. The script also prints a per-sample summary line to the terminal so you can see latency, category, and destination at a glance.

**Prereqs:** Docker Desktop (or Docker Engine), Node 18+, a Groq API key (free).

## How the workflow is shaped

12 n8n nodes, three logical phases:

```
[Caller] → POST /webhook/triage
              │
              ├─ 1.  Validate Input
              ├─ 2.  Build Classify Prompt
              ├─ 3.  HTTP → Groq classify + enrich   (JSON mode, temperature 0)
              ├─ 4.  Parse + Route + Decide Escalation
              ├─ 5.  Build Summary Prompt
              ├─ 6.  HTTP → Groq summarize           (plain text, temperature 0.2)
              ├─ 7.  Assemble Final Record ──┐
              │                              │
              ├─ 8.  Respond to Webhook ◄────┤  returns record to Caller
              │                              │
              └─ 9.  Switch by escalation ◄──┘
                       │
                       ├─► HTTP → Webhook.site/normal      (not escalated)
                       └─► HTTP → Webhook.site/escalation  (escalated)
```

The classification + enrichment fields are produced by **one** Groq call (chosen over two for latency + cost — see [docs/prompt-rationale.md](docs/prompt-rationale.md)). The summary is a second cheap call so its prose can evolve independently. **Routing and escalation decisions are deterministic JavaScript, not LLM calls** — auditable, testable, version-controlled.

For the full design — including the routing table, the escalation rules, the production-scale trade-offs, and the Phase 2 roadmap — read [docs/architecture.md](docs/architecture.md).

## What I'd do differently with more time

In rough order of value:

1. **A separate `disputed_amount` field in the classifier output.** The current routing code infers billing disputes with a regex + `max − min` heuristic that worked for all 5 samples but is fragile. Letting the LLM extract the dispute directly would remove the heuristic.
2. **Calibrate the self-reported confidence.** Hand-label ~100 cases, plot model confidence vs accuracy. If poorly calibrated, replace self-reporting with logprobs or a trained calibrator. Today the 0.70 escalation threshold is well-chosen for our 5 samples but unproven at scale.
3. **A retry + dead-letter loop on Groq.** The 5xx/429 happy path is fine for a free-tier demo; in production you'd want exponential backoff + a DLQ for permanent failures.
4. **Per-destination summary personas.** The Engineering summary should lead with the error code, the Billing summary with the invoice number, the Product summary with the customer's intent. One prompt with branching wouldn't be expensive.
5. **An active-learning loop.** When a human re-routes an escalated record, capture the correction and feed it into a few-shot bank that updates the classifier prompt nightly.

The biggest surprise from this build: **the LLM was rarely the bug**. Sample #1 classified wrong once (fixed with one prompt rule + one few-shot example). Sample #3 looked like a classifier miss but was actually a bug in our deterministic routing code. Worth the explicit mention because the assessment §7 asks about exactly this — see [docs/prompt-rationale.md § "What the AI got wrong that I had to fix"](docs/prompt-rationale.md#what-the-ai-got-wrong-that-i-had-to-fix) for both stories.
